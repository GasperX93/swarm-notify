Swarm Notify Interactive Demo

Encrypted notifications and messaging on Swarm — no servers, no cloud, no accounts.

How discovery works: If you already know someone's ETH address (shared via chat, website, QR code), you can look up their public keys on Swarm and start messaging immediately — no blockchain needed. The on-chain notification registry (steps 6) is only for first-contact discovery: when a stranger sends you a message and you need a way to find out they exist. Think of it like email vs. a phone book — you can email someone directly if you have their address, but the phone book helps strangers find you.

What's Published Where — Privacy Model

Swarm Notify uses encryption for privacy, not access control. Here's exactly what each network sees.

Identity
{ walletPublicKey, beePublicKey, overlay }		 Swarm None — plaintext Anyone who knows your ETH address Your public keys and overlay. Public by design — same model as PGP public keys.

[nonce (12B) | AES-256-GCM ciphertext]		 Swarm ECDH + AES-256-GCM Only sender + recipient That a feed exists between two overlays, and the encrypted blob size. Content is unreadable.

1

Publish Identity (on Swarm)

All scenarios
What: Writes Alice's public keys (for encryption) and overlay address (for feed routing) to a Swarm feed. The feed is at a deterministic topic: keccak256("swarm-identity-" + ethAddress).

Why: So anyone who knows Alice's ETH address can look up her encryption keys on Swarm. This is the foundation — without published keys, nobody can encrypt messages to you.

Data published (plaintext): { walletPublicKey: "02ab...", beePublicKey: "04cd...", overlay: "ff12..." }
Written to Swarm feed at topic keccak256("swarm-identity-" + ethAddress).

Privacy: Public by design — anyone who knows your ETH address can read this. Your public key must be public for others to encrypt data for you (same as PGP). The actual messages are encrypted separately (step 4).
3

Discover Bob (on Swarm)

All scenarios
What: Alice reads Bob's identity feed from Swarm using his ETH address. She now has Bob's public key and overlay address — everything needed to send him encrypted messages.

How does Alice know Bob's address? The same way you know someone's email: it was shared out-of-band — in a chat, on a website, via QR code, or in a contact directory. In this demo, both addresses are visible on screen.

What this enables: With Bob's public key, Alice can derive a shared encryption secret via ECDH (Elliptic Curve Diffie-Hellman). No key exchange server needed — both parties independently compute the same secret from their own private key + the other's public key.
4

Send Message to Bob (on Swarm)

All scenarios
What: Alice encrypts a message and stores it on Swarm in her mailbox feed (Alice→Bob).

Encryption (3 steps): 1. ECDH — Alice's private key + Bob's public key = 32-byte shared secret. Bob can independently compute the same secret (his private key + Alice's public key). 2. AES-256-GCM — Encrypts the message JSON with the shared secret + random 12-byte nonce. GCM also authenticates the ciphertext (tamper detection). 3. Upload — Encrypted blob uploaded to Swarm, then Alice's feed pointer is updated.

Data published (encrypted): [nonce (12 bytes) | AES-256-GCM ciphertext]
Written to Swarm feed at topic keccak256(aliceOverlay + bobOverlay + "swarm-notify").

Privacy: Only Alice and Bob can decrypt the content. An observer can see that a feed exists between two overlays and the blob size, but cannot read the messages. The encryption key is never transmitted — both parties derive it independently.
6

Notify Bob (On-Chain)

Stranger sharing, Public inbox
What: Sends an on-chain notification so Bob can discover that Alice messaged him — without Bob knowing Alice's address in advance.

When is this needed? Only for first contact. If Bob already knows Alice (has her in contacts), he can read her feed directly — no notification needed. But if Alice is a stranger, Bob has no way to know she sent him a message. The on-chain notification solves this: it's a "hey, check your mailbox from me" signal.

Data published (on-chain, encrypted): Notification(keccak256(bobAddress), eciesEncryptedBlob)
The encrypted blob contains: { sender: "0xAlice...", overlay: "ff12...", feedTopic: "a3b2..." }

Privacy: Only Bob can decrypt the payload (ECIES with his private key). An observer on Gnosis Chain can see that someone sent a notification to keccak256(bobAddress), but cannot see who sent it — the sender's identity is inside the encrypted payload. The recipient's raw ETH address is also hidden behind the keccak256 hash (though it's derivable if you already know the address).

Cost: ~22,000 gas (~0.00002 xDAI). Querying is free.

Wallet: Requires xDAI on Gnosis Chain. Paste a funded private key in the "Funded Key" config field above. In production, apps use the Bee node's wallet (already funded for postage stamps).
8

Check Inbox (on Swarm)

All scenarios
What: Alice reads Bob's mailbox feed (Bob→Alice) — the reverse direction from step 4. Same ECDH shared secret derivation, same AES-256-GCM decryption.

Full circle: Alice and Bob are now having an encrypted conversation using only Swarm feeds and math — no server relayed the messages, no authority issued the keys, and no one else can read the content.

Discover Alice (on Swarm)

All scenarios
What: Bob reads Alice's identity feed — same as step 3 but in reverse. Both now have each other's public keys.

What this means: Alice and Bob can now communicate with end-to-end encryption using only Swarm — no blockchain, no server, no key exchange protocol. Steps 1-5 work entirely on Swarm. The on-chain part (step 6) is only needed when a stranger contacts you and you need to discover they exist.
5

Read Messages from Alice (on Swarm)

All scenarios
What: Bob reads Alice's mailbox feed (Alice→Bob) from Swarm and decrypts the messages.

Decryption: Bob computes the same shared secret that Alice used — his private key + Alice's public key produces the identical 32-byte key (ECDH is symmetric). AES-256-GCM decrypts the ciphertext. If the key is wrong or the data was tampered with, decryption fails (GCM authentication).

Key insight: Neither party ever transmitted the encryption key. Both derived it independently from their own private key + the other's public key. This is the magic of Diffie-Hellman key exchange.
6

Poll Notifications (on-chain)

Stranger sharing, Public inbox
What: Bob queries Gnosis Chain for notifications addressed to him, to discover new contacts who sent him messages.

How: Calls eth_getLogs with topic filter keccak256(bobAddress). For each log, tries to ECIES-decrypt the payload with Bob's private key. Successful decryption reveals the sender's address, overlay, and feed topic. Failed decryption = spam or not for Bob, silently discarded.

When is this needed? Only to discover strangers who contacted you. If Bob already knows Alice (from step 3), he can read her feed directly without polling. The registry is a discovery mechanism, not a message transport.

Cost: Polling is free (read-only). Only sending notifications (step 6) costs gas.
7

Reply to Alice (on Swarm)

All scenarios
What: Bob sends an encrypted reply — same ECDH + AES-256-GCM flow as step 4, but on Bob's own mailbox feed (Bob→Alice).

Two feeds per conversation: Alice→Bob and Bob→Alice are separate Swarm feeds, each owned by the sender. To see the full conversation, merge both feeds and sort by timestamp. This design means each party only writes to their own feed — no shared state, no conflicts.
