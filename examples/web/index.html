Swarm Notify Interactive Demo

Encrypted notifications and messaging on Swarm — no servers, no cloud, no accounts.

How discovery works: If you already know someone's ETH address (shared via chat, website, QR code), you can look up their public keys on Swarm and start messaging immediately — no blockchain needed. The on-chain notification registry (steps 6) is only for first-contact discovery: when a stranger sends you a message and you need a way to find out they exist. Think of it like email vs. a phone book — you can email someone directly if you have their address, but the phone book helps strangers find you.

What's Published Where — Privacy Model

Swarm Notify uses encryption for privacy, not access control. Here's exactly what each network sees.

Data Network Encryption Who can read What observers see
Identity
{ walletPublicKey, beePublicKey }		 Swarm None — plaintext Anyone who knows your ETH address Your public keys. Public by design — same model as PGP public keys.
Messages
[nonce (12B) | AES-256-GCM ciphertext]		 Swarm ECDH + AES-256-GCM Only sender + recipient That a feed exists and the encrypted blob size. Content is unreadable.
Notifications
Notification(keccak256(addr), eciesBlob)		 Gnosis ECIES (asymmetric) Only the recipient That someone sent a notification to a hashed address. Sender identity is encrypted inside the payload.

Discover Bob (on Swarm)

All scenarios
What: Alice reads Bob's identity feed to get his public key — everything needed for ECDH key exchange.

How does Alice know Bob's address? Shared out-of-band — in a chat, on a website, via QR code, or in a DAO directory.

Discover Alice (on Swarm)

All scenarios
What: Bob reads Alice's identity feed. Both now have each other's public keys — ready for E2E encryption.

No blockchain needed: Steps 1–6 work entirely on Swarm. On-chain steps (7–8) are only for first-contact discovery.
5

Send Message to Bob (on Swarm)

All scenarios
Encryption: 1. ECDH — Alice's private key + Bob's public key = shared secret. 2. AES-256-GCM — Encrypts with shared secret + random nonce. 3. Upload — Encrypted blob to Swarm, feed pointer updated.

Data published: [nonce (12B) | AES-256-GCM ciphertext] at topic keccak256(aliceAddr + bobAddr + "swarm-notify").

Privacy: Only Alice and Bob can decrypt. The key is never transmitted.

Read Messages from Alice (on Swarm)

All scenarios
Decryption: Bob computes the same shared secret (his private key + Alice's public key = identical key). AES-256-GCM decrypts. Neither party ever transmitted the key — this is Diffie-Hellman key exchange.

Notify Bob (on-chain)

Stranger sharing, Public inbox
When needed: Only for first contact — when Bob doesn't know Alice exists. If they already know each other, skip to step 9.

Data (on-chain, encrypted): Notification(keccak256(bobAddr), ECIES({ sender: "0xAlice..." }))

Privacy: Only Bob can decrypt. Observer sees a notification to a hashed address but cannot see who sent it.

Cost: ~22k gas (~0.00002 xDAI). Requires funded key.

8

Poll Notifications (on-chain)

Stranger sharing, Public inbox
How: eth_getLogs for events matching keccak256(bobAddress). ECIES-decrypt each — reveals sender's ETH address. Bob derives the feed topic. Failed decryption = spam, silently discarded.

Cost: Polling is free (read-only). Only sending notifications costs gas.

Reply to Alice (on Swarm)

All scenarios
Two feeds per conversation: Alice→Bob and Bob→Alice are separate Swarm feeds, each owned by the sender. Merge by timestamp for the full thread. No shared state, no conflicts.
